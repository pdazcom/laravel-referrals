# Laravel Referrals 2.x Product Priorities

## Purpose

This document turns the current 2.x research and release history into an execution-facing product view:

- what problem 2.x is solving
- what should be prioritized next
- what should stay out of scope
- what level of clarity is required before new work enters delivery

It is intentionally short. The goal is focus, not a large roadmap artifact.

## Product Positioning

Laravel Referrals is strongest as referral infrastructure for Laravel applications:

- referral link and code generation
- referral attribution and relationship creation
- event-driven reward triggering
- lightweight production safeguards

It is not currently a full affiliate platform, analytics suite, fraud system, or mobile attribution SDK.

That boundary should stay explicit across all 2.x planning.

## Core Product Goal for 2.x

Make referral attribution and reward triggering reliable across the most common Laravel integration flows, while keeping policy, analytics, and operations in the host application.

## Current State

`v2.3.0` and `v2.4.0` materially improved the product:

- human-friendly referral codes are first-class
- link and manual-code entry flows are documented
- signup and first-purchase hooks exist
- duplicate reward prevention exists
- self-referral prevention exists
- fixed reward examples exist

This means the main 2.x question is no longer "can the package do referral basics?" It is "which adjacent needs should 2.x support without breaking the product boundary?"

## Prioritization

### Priority 1: API-first attribution compatibility

Problem:
The package still reads as web-cookie-first, while real integrations increasingly use APIs, SPAs, and mobile clients.

Why it matters:
- this is the highest-value expansion of the current package boundary
- it keeps the package relevant to modern Laravel architectures
- it improves compatibility without requiring a full new domain model

In scope:
- explicit attachment flows that do not depend on browser middleware
- docs and examples for controller, API, and service-layer attribution
- preserving one simple resolved identity: `ReferralLink`

Out of scope:
- package-owned mobile SDK behavior
- deferred deep linking
- install attribution
- confidence scoring or attribution provenance models

Definition of ready:
- the proposed work explains the attribution input
- the proposal shows how that input resolves to a `ReferralLink`
- the proposal keeps attachment and reward handling separate from transport details

### Priority 2: Hardening and extension seams around package invariants

Problem:
As reward hooks expand, applications need safer defaults and clearer extension points.

Why it matters:
- protects correctness in production
- reduces integration risk without creating a broader fraud or reward platform

In scope:
- relationship-validity hardening
- replay-safe and idempotent reward triggering
- narrow hooks for application-side validation or reward eligibility checks
- explicit skip reasons and observability

Out of scope:
- fraud scoring
- approval workflows
- reversal and clawback systems
- policy builders

Definition of ready:
- the proposal protects a package-owned invariant
- the proposal does not require the package to decide business risk or operator policy

### Priority 3: Reporting clarity, not reporting features

Problem:
Users will ask reporting questions, but the package does not own a reporting product.

Why it matters:
- this is a real user need
- the risk is accidental scope expansion into dashboards and analytics features

In scope:
- docs that define stable reporting inputs
- examples based on package-owned entities and timestamps
- guidance for application-owned instrumentation

Out of scope:
- package dashboards
- analytics APIs
- warehousing or ROI tooling
- finance-grade payout reporting

Definition of ready:
- the work is documentation or a narrow extension seam
- the work relies on package-owned records already present in the model

## Not a 2.x Priority

The following should be challenged before entering active execution:

- configurable attribution engines such as first-click vs last-click rule builders
- package-owned reward lifecycles with pending, approved, reversed, or paid states
- fraud detection platforms
- operator moderation or support override tooling
- package-owned cross-channel analytics products
- mobile install matching and deferred deep-link recovery

These may be valid future opportunities, but they are not good 2.x execution targets without a materially larger product and architecture decision.

## Scope Control Rules

New work should not enter delivery unless it answers all of these clearly:

1. What user or integrator problem is being solved?
2. Why does this belong in package core instead of application code or documentation?
3. Which package-owned concept is changing: attribution input, referral resolution, relationship attachment, reward trigger, or observability?
4. What is explicitly out of scope?
5. What existing 2.x boundary document does the proposal align with?

If those answers are weak, the work is not ready.

## Recommended Near-Term Sequence

1. Prioritize explicit API-first attribution patterns and examples.
2. Add only narrow hardening or extension seams that protect package invariants.
3. Improve reporting guidance through docs, not product surface area.
4. Reject or defer platform-style requests unless the project is ready for a 3.x-level architecture discussion.

## Decision Filter

Use this filter for any new feature request:

- Accept when it makes attribution or reward triggering easier, safer, or clearer within the current package boundary.
- Defer when it requires the package to own policy, operations, analytics, or platform-specific transport complexity.

## Immediate Product Recommendation

If only one product track can move next, it should be API-first attribution compatibility.

Reason:
- highest relevance to modern adoption
- strongest alignment with the existing referral model
- lower scope risk than analytics, fraud, or reward-platform work
- creates a cleaner foundation for future integrations without overcommitting the architecture
